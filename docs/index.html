HDFS, explained

A fact-based tour of the storage layer that made "big data" practical — the ideas that matter, how they actually work, and why each one exists. Every concept here has a matching hands-on lab.

1. Why HDFS exists

HDFS stores very large files reliably across a cluster of ordinary, failure-prone machines, and streams them at high throughput to data-processing jobs.

It was designed (following Google's GFS paper) around a specific, deliberate set of assumptions. Those assumptions explain almost every design choice that follows:

Hardware fails, constantly. With thousands of disks, something is always broken. Fault tolerance must be built into the software, not bought in the hardware.
Files are huge. Gigabytes to terabytes each. The system is tuned for a modest number of large files, not billions of tiny ones.
Write once, read many. Data is typically written once (or appended) and then read many times by batch jobs. No need for random in-place edits.
Throughput > latency. Scanning a whole dataset fast matters more than fetching one record quickly. HDFS optimises for streaming bandwidth.
Move computation to data. Shipping a small program to where the data lives beats dragging terabytes across the network to the program.
Commodity scale-out. Grow capacity by adding cheap nodes, not by buying a bigger, pricier server.
If your workload wants low-latency random reads, tiny files, or frequent in-place updates, HDFS is the wrong tool — see §11.

2. Architecture: NameNode & DataNodes

HDFS uses a master/worker split with two kinds of process:

  • NameNode (master) — keeps all metadata: the directory tree, file names, permissions, and the map of which blocks make up each file and which DataNodes hold them. It holds this in RAM for speed. It never stores file data.
  • DataNode (worker) — stores the actual block data on local disks and serves it to clients. There are many DataNodes; each periodically tells the NameNode it is alive (heartbeat) and what blocks it holds (block report).
The NameNode knows where everything is; DataNodes hold the bytes. Clients get block locations from the NameNode, then transfer data directly with the DataNodes.
Because the NameNode holds the whole namespace in memory, RAM on the master is the practical limit on how many files/blocks a cluster can track — one reason HDFS dislikes tiny files (see §11).

3. Blocks

HDFS does not store a file as one contiguous object. It splits every file into fixed-size blocks (default 128 MiB) and stores each block independently. A 350 MiB file becomes three blocks (128 + 128 + 94).

Why blocks are so large

  • Minimise seek overhead. With big blocks, disks spend nearly all their time streaming data, not seeking. Transfer time dominates; seek time becomes negligible.
  • Keep metadata small. Fewer, larger blocks means fewer entries for the NameNode to track in RAM.
  • Simplify the model. Blocks — not files — are the unit of storage and replication, which makes placement and recovery uniform.
Only the last block is partially full — HDFS does not pad it to the full block size.

4. Replication & rack awareness

Durability comes from replication: every block is copied to several DataNodes — the replication factor, 3 by default. Lose a disk or a whole machine and every block still exists elsewhere. The NameNode continuously watches replica counts and fixes any shortfall.

Replication factor 3: three independent copies of every block.

Rack awareness

A cluster is organised into racks (machines sharing a switch/power). If all three replicas sat in one rack, a rack failure would lose the block; spreading them costs cross-rack bandwidth. The default block placement policy balances the two:

  • 1st replica → the writer's local node (or a random node).
  • 2nd replica → a node on a different rack.
  • 3rd replica → a different node on that same second rack.

Result: survive a whole-rack failure, while only paying to cross the network once.

One replica local, the other two together on a second rack: rack-failure-safe, one cross-rack hop.

5. The write & read paths

Writing — a replication pipeline

The client asks the NameNode to create the file and allocate a block; the NameNode returns a list of DataNodes for the replicas. The client streams the block to the first DataNode, which forwards it to the second, which forwards it to the third — a pipeline. Acknowledgements flow back down the pipeline. The client's upstream bandwidth is used once, not three times.

The write pipeline: the client sends each block once; DataNodes chain the copies.

Reading — from the nearest replica

To read, the client asks the NameNode for the block locations, then connects directly to a DataNode holding each block — preferring the closest one (same node > same rack > another rack). The NameNode is out of the data path entirely, so it never becomes a bandwidth bottleneck.

Metadata from the NameNode; bytes straight from the closest DataNode.

6. Fault tolerance & self-healing

Every DataNode sends a heartbeat to the NameNode every few seconds. If the NameNode stops hearing from a node (default ~10.5 minutes), it declares the node dead. Every block that lived there is now under-replicated, so the NameNode schedules re-replication: it instructs surviving DataNodes to copy those blocks until the replication factor is met again — with no human involvement.

A dead node's blocks are copied onto a survivor until full replication is restored.

Safemode

On startup the NameNode enters safemode — a read-only state. It loads its metadata and waits for DataNodes to report enough block replicas before it will serve writes. This prevents the NameNode from wrongly "fixing" replication before it has heard from the cluster.

7. Metadata durability: fsimage + edit log

The namespace lives in the NameNode's RAM for speed, but must survive restarts. HDFS persists it with two files, not one:

  • fsimage — a checkpoint: a full snapshot of the namespace at a point in time.
  • edit log (edits) — an append-only journal of every change made since that snapshot.

Rewriting a full snapshot on every change would be far too slow, so the NameNode just appends each change to the edit log. On restart it loads the last fsimage and replays the edit log to reconstruct the current state. Periodically a checkpoint merges the two into a fresh, larger fsimage so the edit log doesn't grow without bound (done by a Secondary or Standby NameNode, or by hand with hdfs dfsadmin -saveNamespace).

Snapshot + journal = the classic durable-metadata pattern (like a database).

8. Interfaces: the FS shell & WebHDFS

The everyday interface is the FS shell, a familiar set of POSIX-like commands:

hdfs dfs -mkdir -p /data/input
hdfs dfs -put local.csv /data/input/
hdfs dfs -ls  /data/input
hdfs dfs -cat /data/input/local.csv
hdfs dfs -get /data/input/local.csv ./copy.csv
hdfs dfs -rm  /data/input/local.csv

For programs that don't speak HDFS's native RPC, WebHDFS exposes the whole filesystem as an HTTP REST API — usable from curl or any language:

# list a directory as JSON
curl 'http://namenode:9870/webhdfs/v1/data/input?op=LISTSTATUS'
# create a file (the NameNode 307-redirects the upload to a DataNode)
curl -L -X PUT -T local.csv 'http://namenode:9870/webhdfs/v1/data/input/local.csv?op=CREATE'

9. Quotas, trash & permissions

Quotas. A directory can be capped by a name quota (max number of files+dirs) and/or a space quota (max bytes, counting replication). Exceeding either makes the write fail — a guardrail so one team can't fill the cluster.
Trash. With fs.trash.interval>0, hdfs dfs -rm moves a file to a per-user .Trash folder instead of deleting it, so mistakes are recoverable until it's purged. -skipTrash forces immediate deletion.
Permissions. HDFS has a POSIX-like model (owner/group/other, r/w/x) plus optional ACLs — enough for multi-tenant access control.
Snapshots. A directory can be snapshotted for point-in-time, read-only recovery of its contents — cheap because it only records differences.

10. Availability & scaling out

High Availability (HA)

A single NameNode is a single point of failure: if it dies, the whole filesystem is unavailable. HA runs two NameNodes — one active, one standby:

  • A JournalNode quorum holds a shared edit log; the active writes to it, the standby reads from it, staying in sync.
  • ZooKeeper plus a ZKFC (failover controller) beside each NameNode detect the active's death and automatically promote the standby. Fencing ensures the old active can't keep acting (no "split brain").
Active/standby NameNodes stay in lock-step through the shared journal; ZooKeeper + ZKFC handle automatic, fenced failover.

Federation

One NameNode's RAM caps how much namespace it can hold. Federation runs several independent NameNodes, each owning a slice of the namespace, all sharing the same pool of DataNodes — scaling metadata capacity horizontally. (HA and Federation are orthogonal and often combined.)

11. Limitations: when NOT to use HDFS

Anti-patternWhy it hurts
Millions of tiny filesEach file/block is an in-RAM entry on the NameNode. Many small files exhaust the master's memory long before they fill the disks. Bundle them (e.g. sequence/Har/Parquet files) instead.
Low-latency single-record readsHDFS optimises for throughput on large streaming reads, not millisecond point lookups. Use a database/key-value store (e.g. HBase on top of HDFS) for that.
Frequent in-place editsHDFS is write-once / append-only. There is no random in-place update of file contents.
Many concurrent writers to one fileA file has a single writer at a time; HDFS is not built for many-writer concurrency to the same file.
